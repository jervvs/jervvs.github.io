I apply to fairly different roles — data engineering, platform, AI-for-SWE, IT ops. For a while I kept a separate résumé per role and let them drift out of sync. That's a maintenance trap: fix a date or a metric in one, forget the other four. So I switched to a system that treats a CV like code with a single source of truth.

## One evidence bank, many cuts

There's exactly one master document — the **superset of everything I've genuinely done**, every bullet, every metric, in full. I never send it. It's the source of truth. Each application gets a *cut*: a role-specific résumé built by selecting and reordering from that bank. Change a fact once, in the master, then propagate to the cuts. No drift.

The rule that keeps this honest: **I select and reframe; I never invent.** Job titles stay truthful. What changes between cuts is emphasis and vocabulary, not substance.

## Tailoring per application

For each posting, four moves:

1. **Mirror the JD's exact language.** Screening runs on two technologies — literal keyword matching (legacy ATS) and semantic embedding similarity (newer AI screeners). You cover both by saying the true thing clearly *and* using the posting's own phrasing. Include the acronym and the spelled-out form once — "CI/CD (Continuous Integration / Continuous Delivery)" — because it costs nothing and hits both.
2. **Reorder so the most relevant bullets lead.** Same evidence, resequenced. The reviewer reads top-down; put the bullets that match this JD first.
3. **Rewrite the summary and skills per role.** These are the highest-signal, most keyword-dense sections. They get retuned every time; the experience bullets mostly just reorder.
4. **Tailor per posting — every time.** A single generic CV blasted everywhere is now *actively* risky: when many employers use the same AI vendor, one poorly-matched résumé can get filtered out everywhere at once ("algorithmic monoculture"). Genuine relevance per application is the whole game.

## Why this beats a single polished CV

A great generic CV optimizes for no one. The evidence-bank approach lets me be simultaneously honest (one truth) and relevant (framed for the reader in front of me) — without maintaining five diverging documents by hand.
